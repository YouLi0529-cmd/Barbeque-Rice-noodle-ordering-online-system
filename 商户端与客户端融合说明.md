# 商户端与客户端融合说明

当前 `main` 已经以赵浩然的客户端分包结构为主体，并把你的商户端融合为独立 admin 分包。

## 入口结构

- 用户端首页：`pages/covertest/covertest`
- 堂食点单：`packages/order/pages/index/index`
- 露营点单：`packages/camping/pages/campingorderfood/campingorderfood`
- 用户订单：`packages/user/pages/myorder/myorder`
- 商户后台：`packages/admin/pages/admin/admin`

商户后台不在用户端页面里显示入口。开发阶段可以在微信开发者工具的“编译模式”里选择“商户后台入口”，或直接编译路径：

```text
packages/admin/pages/admin/admin
```

正式使用时，建议只给店员内部使用这个路径生成的小程序码/收藏入口，不在顾客首页放可见按钮。后台真正的安全仍然要依赖管理员密码和后续权限校验。

## 桌码路径

桌码生成现在指向赵浩然的新点单页：

```text
packages/order/pages/index/index
```

也就是说顾客扫码桌码后，会进入堂食点单页，并通过 `scene` 参数带入桌号。

## 订单付款逻辑

客户端提交订单后，订单默认是线下未付款：

```js
pay_status: false
payStatus: false
payMethod: 'offline'
status: 'waiting_pay'
frontDeskConfirmed: false
kitchenPrinted: false
kitchenPrintStatus: 'pending'
```

商户端的两个动作互相独立：

- `发送厨房`：前台确认点菜单后打印厨房单，不改变付款状态。
- `确认付款`：顾客线下付款后前台手动确认，只改变付款状态。

## 后续协作边界

- 赵浩然主要改 `packages/order`、`packages/camping`、`packages/user` 和客户端相关云函数。
- 你主要改 `packages/admin` 和商户端相关云函数。
- 双方都不要随意改订单核心字段名：`goods`、`status`、`pay_status`、`payStatus`、`frontDeskConfirmed`、`kitchenPrinted`。
- 如果要新增业务字段，优先新增字段，不要改变旧字段含义。
